# Self-hosted VPN & secure tunnel toolkit (AmneziaWG + VLESS/REALITY)

Тулкит для развёртывания **собственной** приватной VPN-инфраструктуры: узел AmneziaWG
(WireGuard с обфускацией трафика) с защищённой веб-панелью и опциональная прослойка VLESS+REALITY
с региональным релеем — для приватного, шифрованного и устойчивого доступа к **своим** серверам и
сервисам из разных сетевых окружений.

> **Назначение и ответственность.** Проект предназначен для построения собственной приватной сети
> и защищённого доступа к ресурсам, которыми вы владеете или к которым у вас есть правомерный доступ
> (свои серверы, домашняя/офисная сеть, личные сервисы за рубежом, безопасность на недоверенном
> Wi-Fi). Он **не предназначен** для нарушения закона или доступа к ресурсам, ограниченным в
> соответствии с применимым законодательством. Ответственность за законность использования несёт
> пользователь; соблюдайте законодательство своей юрисдикции (в т.ч. законодательство РФ).

## Сценарии
- Защищённый удалённый доступ к собственным сервисам с публичных/недоверенных сетей.
- Шифрование трафика на чужом Wi-Fi.
- Связность между своими географически распределёнными узлами (site-to-site / точка-точка).
- Устойчивый канал к своей инфраструктуре в сетях с жёсткой фильтрацией исходящих соединений
  (корпоративные egress-политики, captive-порталы), где обычные VPN-протоколы нестабильны.

## Архитектура

```
                 ┌── базовый VPN ──────────────────────────────────────────────┐
клиент ─AmneziaWG─────────────────────────────────────────────▶ exit-узел ───▶ свои ресурсы / интернет
                 └─────────────────────────────────────────────────────────────┘

                 ┌── транспорт через региональный релей ───────────────────────┐
клиент ─VLESS+REALITY(TLS:443)─▶ региональный релей (тонкий, не расшифровывает)
        ─(шифрованный туннель)─▶ exit-узел: REALITY терминируется ──▶ свои ресурсы
                 └─────────────────────────────────────────────────────────────┘
```

## Прослойка 1 — AmneziaWG (база)

### Сервер: `deploy-awg-portal.sh`
Разворачивает на чистой Ubuntu стек за Caddy + Authelia:
- **amnezia-wg-easy** — узел AmneziaWG + веб-панель (userspace), наружу только `51820/udp`.
- **Caddy** — реверс-прокси с авто-TLS; apex и голый IP отдают нейтральную заглушку.
- **Authelia** — портал входа с паролем + 2FA (TOTP), единственный гейт к панели.

```bash
curl -fsSL "https://github.com/Motokichirou/selfhosted-VLESS-AmneziaWG-autodeploy/raw/refs/heads/main/deploy-awg-portal.sh" -o deploy-awg-portal.sh
sudo bash deploy-awg-portal.sh
```
Интерактивный (спрашивает домены/IP) — качай файл, не запускай через `curl | bash`. Идемпотентен;
том ключей `~/.amnezia-wg-easy` сохраняется.

### Клиент MikroTik: `awg-client-mikrotik.rsc`
RouterOS 7.x: контейнер `awg-proxy` + **policy-based routing** (заданные адреса идут в туннель,
остальное — напрямую). Адресный список настраиваемый. Вставляешь `.conf` из панели в блок CONFIG,
заливаешь, `/import file=awg-client-mikrotik.rsc`. Детали — в шапке `.rsc`.

## Прослойка 2 — VLESS + REALITY (через региональный релей)

Опциональная прослойка для **устойчивой связности** в сетях, где обычные VPN-протоколы (UDP/без TLS)
нестабильны. Транспорт — **VLESS+REALITY поверх TCP:443**: соединение выглядит как обычный HTTPS к
популярному сайту (REALITY-камуфляж). Региональный релей служит точкой входа и туннелирует трафик к
твоему exit-узлу, где REALITY терминируется. Релей **тонкий** — ничего не расшифровывает.

### Компоненты
- **`YANDEX-RELAY-SETUP.md`** — провижининг релей-ВМ (Yandex Cloud), установка AmneziaWG, туннель до exit-узла.
- **`relay-forward-setup.sh`** — на релее: проброс входящего `:443` внутрь туннеля + персист.
- **`deploy-3xui.sh`** — на exit-узле: панель **3X-UI** (Xray) за тем же Caddy + Authelia, REALITY-inbound.
- **`REALITY-SETUP.md`** — настройка REALITY-inbound (выбор камуфляж-домена, поля, клиентская ссылка).
- **`Yandex-Cloud.conf.example`** — шаблон конфига релея.

### Камуфляж-домен (REALITY)
REALITY маскирует соединение под обычный TLS к выбранному домену. В качестве цели берётся
стабильный широко используемый домен с поддержкой **TLS 1.3 + HTTP/2 + X25519** (крупный CDN/статик-хост,
например `yastatic.net`). Это штатный механизм REALITY, а не «обход» — просто выбор камуфляжа.

### Порядок развёртывания (прослойка 2)
1. `YANDEX-RELAY-SETUP.md` — поднять релей + туннель до exit-узла.
2. На exit-узле: `sudo bash deploy-3xui.sh` (нужна A-запись поддомена) → создать REALITY-inbound.
3. `relay-forward-setup.sh` на релее + стыковочный DNAT (`REALITY-SETUP.md`).
4. Собрать клиентскую `vless://`-ссылку + QR. Клиенты: iOS — Happ/Streisand; Android — v2rayNG/husi.

## Безопасность
- Заглушка на apex/голом IP — снижение видимости для случайных сканеров, не замена защиты. Реальную
  защиту держат TLS + сильный пароль + 2FA + свежий софт.
- Релей завязан на личность владельца и его память видна гипервизору — поэтому REALITY и egress
  держим на своём exit-узле, а релей делаем тонким (знает только про exit-узел). Хардening релея —
  в `YANDEX-RELAY-SETUP.md` §5.
- **Секреты не коммитим**: приватные ключи, реальные `.conf`, доступы — в `.gitignore`. Публикуется
  только код и обезличенные гайды.

## Ограничения
Туннель не помогает, если нижележащий канал физически недоступен или сильно урезан по полосе —
трафик идёт по той же трубе.
